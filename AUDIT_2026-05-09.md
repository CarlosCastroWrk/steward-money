### Three User Journeys — End-to-End

**Journey 1: User taps Sync on Dashboard**

1. `components/dashboard/DashboardSyncButton.tsx` renders a button; on tap calls `syncNow()` from `hooks/useAutoSync.ts`
2. `useAutoSync.syncNow()` POSTs to `/api/plaid/sync`
3. `app/api/plaid/sync/route.ts` — authenticates user via `createClient()` + `supabase.auth.getUser()`
4. Fetches all `plaid_items` for user; for each item: calls `plaidClient.accountsBalanceGet()` (live, not cached), upserts account balances; then calls `plaidClient.transactionsSync()` in a cursor-based loop, upserts posted/pending transactions, removes stale pending rows, deletes removed transactions
5. Updates `plaid_cursor` on each `plaid_items` row
6. Calls `autoDetectBillPayments()` on new transactions
7. Creates `alerts` for large deposits (>$200)
8. Returns `{ accounts_updated, transactions_synced, item_errors? }`
9. `useAutoSync` receives OK → writes timestamp to `localStorage` → calls `onSyncComplete()` → `router.refresh()` reloads server component data → `toast("Synced — balance updated.")`

**Journey 2: User sends a message to Luka**

1. User types in `components/Luka.tsx` textarea; on submit, `sendMessage()` fires
2. Optimistically renders user message; POSTs to `/api/luka` with last 20 messages as context
3. `app/api/luka/route.ts` — auth check → rate limit check (20 req/min per user)
4. Parallel context fetch: `calculateSafeToSpend`, alerts, Silas insights, Solomon word, Kairos pending, agent memories, connected accounts, calendar events
5. Builds full system prompt with live financial snapshot
6. Anthropic `messages.create()` loop (up to 6 iterations) using `claude-sonnet-4-6` with 24 tools defined
7. If `stop_reason === "tool_use"`: executes tool (DB reads/writes), appends tool results, loops
8. If `stop_reason === "end_turn"`: logs usage to `agent_usage`, returns `{ reply, refreshNeeded, actions }`
9. Client saves both messages to `luka_conversations`; if `refreshNeeded`, calls `router.refresh()` and dispatches `financials:changed` event
10. After 2nd user message, calls `/api/luka/title` to generate a conversation title via `claude-haiku-4-5-20251001`

**Journey 3: User connects bank via Plaid**

1. `PlaidLinkButton` (or `ConnectBankCard`) calls `/api/plaid/create-link-token` (POST) → returns `link_token`
2. `react-plaid-link` opens the Plaid Link iframe using the token
3. On success, Plaid calls `onSuccess(public_token, metadata)` → client POSTs `{ public_token, institution_name, institution_id }` to `/api/plaid/exchange-token`
4. `exchange-token` calls `plaidClient.itemPublicTokenExchange()` → gets `access_token` + `item_id`
5. Inserts row into `plaid_items` (access token stored in DB)
6. Calls `plaidClient.accountsGet()` → inserts/upserts all accounts with balance semantics (depository only gets `available_balance`)
7. Calls `plaidClient.transactionsSync()` without cursor (full history, up to 500/page) in a loop, upserts all transactions
8. Persists final cursor to `plaid_items` for future incremental syncs
9. Returns `{ accounts_synced, transactions_synced }`
10. Client shows success toast; if webhook is configured, Plaid sends subsequent update webhooks to `/api/plaid/webhook`

## Section 4: SECURITY REVIEW

### CRITICAL

**None found.** No exposed secrets, no unauthenticated access to user financial data, no SQL injection vectors (all queries use Supabase's parameterized builder, never raw SQL with user input).

### HIGH

**[HIGH] Cron header spoofing — `/api/agents/argus`, `/api/agents/solomon`, `/api/agents/silas`, `/api/agents/nova`, `/api/monitor`, `/api/bills/reset-monthly`**

These routes use `req.headers.get("x-vercel-cron") === "1"` to determine if a request is from Vercel's cron scheduler. An unauthenticated external caller can POST with this header and trigger the cron path — iterating all users' financial data with the admin client, generating AI responses for all users, or resetting all bills' `paid_at` fields.

Vercel Cron does not provide cryptographic verification of this header in the free/hobby tier. On Pro/Enterprise, Vercel signs cron requests with `CRON_SECRET`. The fix is: set `CRON_SECRET` env var, check `Authorization: Bearer ${CRON_SECRET}` header before accepting cron path. File: all routes listed above. Risk: an attacker could trigger mass AI generation (cost attack) or reset payment status for all users.

**Mitigating factor:** Billing happens through Anthropic API key (cost), not direct data exfiltration. RLS still applies to the admin client operations. But resetting `paid_at` for all bills at arbitrary times is a real functional attack.

**[HIGH] Plaid access tokens stored in plain text in `plaid_items` table**

`app/api/plaid/exchange-token/route.ts` line 34 inserts `access_token: accessToken` directly into `plaid_items`. The `app/api/debug/balances/route.ts` fetches `access_token` from this table to call Plaid APIs. This is a standard pattern for Plaid integrations, but access tokens are permanently valid until revoked. If the Supabase database is breached, all users' bank connections are compromised. Recommendation: At scale, encrypt access tokens at rest using a KMS before storing. For a personal app / pilot this is acceptable risk.

### MEDIUM

**[MEDIUM] Google OAuth access token passed through client then stored server-side**

The Google Calendar GSI implicit flow returns `access_token` to the browser (e.g., `CalendarOptInCard.tsx`, `ComingUpWidget.tsx`, `components/settings/CalendarSection.tsx`). The client then POSTs this token to `/api/calendar/connect`. The token is stored in `calendar_connections` as plain text. If a malicious script runs in the browser (XSS), it could intercept the token. Since the app sets `Content-Security-Policy` with `unsafe-inline` in scripts (required for Plaid and Google), CSP provides limited XSS protection. The GSI implicit flow is deprecated by Google in favor of authorization code flow — this should eventually be migrated.

**[MEDIUM] Rate limiter is per-serverless-instance, not global**

`lib/rate-limit.ts` uses an in-memory `Map`. On Vercel, each serverless function invocation may be a different instance with its own clean state. This means the rate limiter is ineffective under moderate load — a user could hit 20 req/min × N cold instances. For a personal app this is acceptable, but the comment "Works per serverless instance — good enough for a personal app" is accurate self-assessment. For production: use Redis or Vercel KV.

**[MEDIUM] `/api/email/welcome` is unauthenticated**

`app/api/email/welcome/route.ts` has no auth check. Any caller with the URL can POST any email address and trigger a welcome email. This can be used to spam arbitrary email addresses. Fix: add auth check or verify the email matches the current session's user email.

**[MEDIUM] `next_date` column accessed on `income_sources`**

Multiple routes query the non-existent (per CLAUDE.md) `next_date` column: `app/api/monitor/route.ts` line 23, `app/api/agents/solomon/strategy/route.ts` line 17, `app/api/agents/silas/route.ts` line 27, `app/api/luka/route.ts` line 317. If the column doesn't exist, these queries silently return null data rather than erroring. This means Silas's post-paycheck surge pattern (line 52 of silas/route.ts: `i.next_expected_date || i.next_date`) degrades silently.

**[MEDIUM] Solomon strategy route references non-existent settings columns**

`app/api/agents/solomon/strategy/route.ts` line 16 selects `giving_pct`, `savings_pct`, `trading_pct`, `kairos_life_stage`, `main_financial_goal` from `user_settings`. These column names don't match the schema (`giving_value`, `savings_value`, `life_stage`, `main_goal`). The query silently returns null for all these fields, meaning Solomon's strategy is generated with empty context.

### LOW

**[LOW] Debug routes accessible to all authenticated users**

`/api/debug/balances` and `app/debug/agents/page.tsx` are accessible to any logged-in user. `debug/balances` fetches Plaid `access_token` and passes it to `plaidClient.accountsGet()` — the access token is used server-side but the route returns detailed account data including live Plaid balances. For a personal app with one user, this is fine. Add admin check before broader rollout.

**[LOW] `calendar_context.ts` contains user-specific hardcoded logic**

## Section 5: PERFORMANCE & RELIABILITY

### Backend

**N+1 Query Patterns:**

`app/api/agents/argus/route.ts` — cron path (line 128–138): fetches all users from `income_sources`, then iterates each user calling `runArgus()` sequentially. Each `runArgus()` call does 7 parallel queries. With 100 users = 700 Supabase calls sequentially. Not a problem at pilot scale (1-5 users) but will be slow at 50+ users. Fix: batch with `Promise.all()` in groups of 10-20.

Same pattern in `app/api/agents/solomon/route.ts` (line 136–144), `app/api/agents/silas/route.ts` (line 139–147), `app/api/agents/nova/route.ts` (line 86–94).

`app/api/plaid/sync/route.ts` lines 28–36: fetches all accounts, then in the bill auto-detect loop (lines 19–35), fetches bills once but loops over all new transactions and all bills in a nested loop — O(transactions × bills). For most users this is <100 × <50, so fine.

**Missing DB Indexes:**

The migrations don't create indexes on `transactions(user_id, date)`, `transactions(plaid_transaction_id)` (used for upsert conflict), `bills(user_id, next_due_date)`, or `alerts(user_id, created_at)`. Supabase auto-creates indexes on primary keys and unique constraints, but not on the high-frequency query patterns. At hundreds of transactions these will matter.

**Routes doing too much:**

`app/api/luka/route.ts` is 973 lines — system prompt construction, all 24 tool implementations, and the agentic loop in one file. The tool execution function `executeTool()` (lines 304-756) is a 450-line switch statement. This works but is hard to test and maintain.

`app/api/agents/council/route.ts` fans out 6 parallel Anthropic API calls plus a synthesis call = 7 Anthropic calls per request. Each costs ~$0.003-0.015. At the 3 req/min rate limit, max cost is ~$0.30/min per user. Fine for personal use.

**Missing Timeouts:**

No timeout wrappers around Anthropic API calls or Plaid API calls. Vercel serverless functions have a 10-second timeout by default (30s on Pro). A slow Anthropic response or Plaid network issue could leave users staring at a loading indicator. The Luka route does up to 6 Anthropic iterations which could exceed 30s on complex tool chains. Recommendation: add `AbortController` with ~25s timeout on the Anthropic call.

**Caching:**

`lib/supabase/server.ts` wraps `createClient` in React `cache()` — correct, deduplicates the client within a single server component render tree. No other caching in use. The `SyncOnFocus` component caches the last sync timestamp in `lastSyncedAt` ref but this resets on page reload (only prevents double-syncing within a session). `useAutoSync` persists to `localStorage`. These are sensible choices.

**Agent Call Frequency and Cost:**

- Luka: triggered by user, up to 6 Anthropic calls per message, ~$0.01-0.05 per conversation turn
- Council: 7 calls per request, ~$0.05-0.10 per request, rate-limited to 3/min
- Solomon: 1 call/week/user, negligible
- Nova: 1 call/day/user via cron
- Calendar auto-sync: 1 Haiku call per user per day (manual sync) or per 15 events (cron uses Sonnet for auto-sync — this is a cost discrepancy vs the manual path which uses Haiku)
- Eden, Iron: triggered manually, low frequency

The `/api/calendar/auto-sync/route.ts` uses `claude-sonnet-4-6` (line 72) for classification during the daily cron, while `/api/calendar/sync/route.ts` correctly uses `claude-haiku-4-5-20251001` for manual sync. The cron path is 12x more expensive per classification without a documented reason.

**Duplicate Sync Logic:**

`SyncOnFocus.tsx` (30-min threshold) and `useAutoSync` hook (2-min cooldown via localStorage) both auto-trigger Plaid sync on tab focus. These can conflict — `SyncOnFocus` fires at 30 min, `useAutoSync` fires at 2 min. The `useAutoSync` cooldown is per-session (localStorage) while `SyncOnFocus` uses an in-memory ref. A fresh page load could trigger both.

### Frontend

**Bundle Dependencies:**

`framer-motion` (^12.38.0) is imported only in `components/SwipeableNavigator.tsx` for page transition animations. This is a large library (~130kb gzipped) used for a single swipe animation. Consider replacing with CSS animations (which already exist in `globals.css` for similar effects).

`recharts` is in `package.json` but **not imported anywhere** in the source code. This adds ~300kb to the dependency tree for nothing.

**`next/image`:** Not used. No images in this app — all UI is text + icons (Lucide). Not a problem.

**Font Loading:** Google Fonts (`Instrument_Serif`) loaded via `next/font/google` with `display: "swap"` — correct. Geist fonts loaded from the `geist` package (no network request). Font strategy is solid.

**Large Components:**

- `components/Luka.tsx` at 1,049 lines contains the full chat UI, speech recognition hook, conversation list, multiple sub-components, and 8 SVG icon components all in one file. Should be split.
- `components/pulse/PulseView.tsx` at 693 lines does all its own data fetching via `useEffect` + `fetch()`. It's essentially a mini page with no server-side data. Could become a server component.
- `components/bills/ExpensesView.tsx` at 632 lines.

**Missing `useMemo`/`useCallback`:**

`Luka.tsx` — `sendMessage` wrapped in `useCallback` (correct). `computeGreeting` and `computeChips` called inline in `MessageList` on every render — these are pure functions based on `lukaContext`, should be memoized. Minor perf issue.

**Route Prefetching:**

`BottomNav.tsx` calls `router.prefetch()` for all 4 primary nav routes in `useEffect` — correct and proactive.
